# sir v0.1.1 — Friction-vs-Security Hardening Plan

> **Scope:** This plan targets the **shipped standalone `sir` at v0.1.1** (synced GitHub `main`, commit `58c808e`). It is the actionable output of a 22-agent read-only audit+design pass (1.67M tokens, 478 tool uses). North star: *block less unless absolutely necessary*, bounded by the CLAUDE.md non-negotiables (std-lib only; never widen a Rust deny; fail-closed; never leak a secret).
>
> **Method:** 8 specialist auditors → synthesis into a blocking budget → adversarial red-team of every proposed downgrade. Nothing here was auto-applied; downgrades that touch behavior require operator sign-off.

## Operator decisions & execution scope (2026-05-28)

The operator reviewed the plan and chose:

- **NET-1 / egress (headline):** *Harden the secret scanner first, then downgrade.* Close the scanner blind spot (URI-with-userinfo / connection strings / decoupled env-read taint / bare-value entropy) so a "clean" session genuinely means no secret in context — **then** downgrade clean-session `net_external`/`dns_lookup` to ask+auto-lease. The downgrade (NET-1/NET-2) is gated on the scanner work (new item **SCAN-1**) landing first.
- **Friction downgrades (all four approved, each behind the profile gradient with the red-team's mandatory guards):** ENV-1, NPX-1 / REMOTE-1 / MCPDRIFT-1, NETALLOW-1, MCPONBOARD-1.
- **Security track:** **OBSERVE-1 only** (exempt `mcp_credential_leak` + operator-choice secret-egress from observe-mode downgrade). **Relay auth (RELAY-1/2/3) deferred** — and with it RELAY-4 (auth-line log sanitizer) and the auth-specific portions of RELAY-5.

**Execution order (dependency-ordered waves, each gated on `make test` green):**

1. **Wave 1 — Safety net + trivial wins (safe-additive, zero behavior change):** DOC-1, FLOOR-1, FLOOR-2, FLOOR-3, BUDGET-1, SECRET-1, DETECT-3, DETECT-4. These pin the floor and current behavior so every later change is provably non-regressing.
2. **Wave 2 — Scanner hardening (pure security gain; NET-1 prerequisite):** SCAN-1 (new), SECRET-3.
3. **Wave 3 — Observe floor (security tightening):** OBSERVE-1.
4. **Wave 4 — Profile gradient (master switch):** PROFILE-1 (preserves personal=deny+redact).
5. **Wave 5 — Friction downgrades (behind profile, guarded):** ENV-1, NPX-1, REMOTE-1, MCPDRIFT-1, NETALLOW-1, MCPONBOARD-1.
6. **Wave 6 — Egress downgrade (after SCAN-1):** NET-1, NET-2.
7. **Wave 7 — Additive value & polish:** DETECT-2 + DETECT-1 (signal_ids), FRICTION-1/FRICTION-2, UX-1/UX-2/UX-3, DOC-2, PROFILE-2.

**Deferred (not this effort):** RELAY-1, RELAY-2, RELAY-3, RELAY-4, RELAY-5 (auth/deployment-doc portions).

## Executive verdict

sir v0.1.1 is closer to the north star than the headline gaps suggest: normal coding is genuinely silent, the redaction/auto-lease machinery already exists, and the narrow-only invariant holds today (applyObserveMode at observe.go:46-55 is the ONLY Go path that turns a Rust Deny into a wire Allow, and it sits behind the integrity guards). The path to "perfect" is therefore not loosening the oracle broadly — it is (a) landing a tier of safe-additive guard tests and harnesses that PIN the floor so friction downgrades can ship without eroding security, then (b) approving a scoped set of needs-signoff downgrades in batches. The single most consequential reconciliation: the security-floor audit's "net_external/dns_lookup deny always" is NOT actually a security floor — its defended floor is secret-exfil and control-plane integrity, and clean-session external egress (no secret, no taint) has no exfil value, only friction. Because the secret/derived-secret egress deny is independently enforced in THREE oracle locations (policy_guards.rs:131-136 + :158-162 + policy_sinks.rs:41-54), flipping only the !session_secret branch of evaluate_network_guardrails to Ask cannot widen any deny — making the headline downgrade needs-signoff-but-provably-safe. I also caught an error the friction-defaults audit carried: it assumed personal profile leaves DenyRawSecretReads off (the "redacted-ask" path), but policy.go:311 sets it true, and the code comment (policy.go:303-309) correctly argues deny+redact is LOWER friction than an ask because it keeps a thinking-enabled turn linear; the profile work item must preserve that, not undo it. The genuinely dangerous downgrade hiding in the codebase is observe mode: applyObserveMode (observe.go:50) currently downgrades EVERY deny including mcp_credential_leak and secret->egress to wire-allow during a rollout — that needs its own signoff decision, separate from the friction items. Net: ship the P0 safe-additive guard layer now; stage the friction downgrades behind a personal/team/strict/managed profile gradient with operator sign-off; treat relay auth and the observe-floor exemption as the two security-critical needs-signoff tracks.

### The one finding that reframes everything

The "obvious" #1 friction win — letting `curl`/`dig` run on a *clean* session — is the **#1 security risk**, and the red-team rejected it. Reason: the clean-session deny is only "safe to remove" if a clean session truly means *no secret in context*. It does not. The credential scanner (`pkg/secretscan/structured_scan.go`) recognizes ~15 fixed prefixes (`AKIA`/`ghp_`/`sk_live_`/…) plus a constrained high-entropy heuristic. It is **blind to** DB connection strings (`postgres://svc:S3cr3tP@ss@host/prod`), hex passwords, OAuth-in-URL, and custom-named config secrets. So an agent can hold a real secret on a session that never trips taint — and downgrading the egress deny opens a confused-deputy exfiltration path for exactly those secrets.

**Consequence for the roadmap:** to *safely* block less on egress, sir must first **detect secrets better**. Friction reduction on network sinks is gated on closing the scanner blind spot, not on flipping the oracle verdict.

## The floor — minimal necessary blocks (confirm these stay)

1. Posture-file write/delete always asks (CLAUDE.md #9) — policy_guards.rs:45-60. Never downgrade.
2. Secret-session egress denies: under session_secret, net_external/dns_lookup/push_remote/delegate -> Deny (policy_guards.rs:149-218). Never downgrade; never widen.
3. Derived-secret-taint egress denies: when content derives from secret data, net_external/dns_lookup/push_remote -> Deny (policy_sinks.rs:5-59). Never downgrade.
4. mcp_credential_leak: raw credential pattern in MCP args to an untrusted server -> Deny (evaluate_preflight.go:17-53). Never downgrade — and exempt from observe-mode downgrade (OBSERVE-1).
5. Control-plane integrity: deny-all / session-integrity / lease-integrity failure -> Deny everything, fail-closed (evaluate_state.go:13-101; post_evaluate_checks.go:38-150; policy_guards.rs:12-25; CLAUDE.md #3/#8). Must run before the observe defer and survive observe mode.
6. Raw-secret-read of a recognized sensitive file under DenyRawSecretReads -> Deny + redacted view (secret_read_gate.go:32-61). The redact-substitute is the substitute; the block stays.
7. Corrupted/unreadable session or lease state fails CLOSED; only os.IsNotExist may seed fresh defaults (session_io.go:14-30, evaluate_io.go:92-104, lease.go:224-234). DefaultLease.ObserveOnly=false so corruption can never seed observe mode.
8. NARROW-ONLY INVARIANT (CLAUDE.md #2): Go may add restrictions but must never turn a Rust Deny into a wire Allow except via the single deliberate, lease-gated applyObserveMode path. Every friction downgrade below lives in Go's narrow space, in lease/config defaults Go owns, or narrows a Rust !session_secret deny to an ask — none widens a deny.
9. SCOPING NOTE (the reconciliation): 'net_external/dns_lookup deny ALWAYS' is NOT a security floor. The defended floor is the secret/taint condition above. Clean-session external egress is friction-only and is the headline downgrade candidate (NET-1/NET-2), pending sign-off.

### ⚠️ DOWNGRADE-WITH-MITIGATION — MCP onboarding gate for ALREADY-APPROVED servers within the 24h / 20-call window (pkg/hooks/evaluate_preflight.go:181-23

**Attack:** Confused-deputy exfil via an approved-but-unscoped MCP server. The MCP server process then exfiltrates `content` out-of-band over its own socket. TODAY the onboarding ask (evaluate.go:204) is the ONLY interactive checkpoint that surfaces this brand-new server's first call to the user; with the downgrade it silent-allows and is merely logged.

**Mitigation (required to ship safely):** Do NOT key the default-disable purely on the burnable per-session counter. ### ✅ SAFE-TO-DOWNGRADE — relay POST /slack/interactions — no Slack signing-secret verification (relay.go:212-252)

**Proposed:** Keep the endpoint reachable but add optional Slack v0 HMAC-SHA256 signature verification gated on SIR_SLACK_SIGNING_SECRET. Default-off preserves today's always-ack-200 behavior; when the operator sets the secret, forged/unsigned requests return 401 (deliberately overriding the always-ack invariant for FORGED requests only).

**Attack:** A prompt-injected (confused-deputy) agent or any host on the relay's network forges a Slack button-click: POST /slack/interactions with form field payload={"actions":[{"value":"<attacker string, e.g. a stolen secret>"}]}. Goal: get the relay to (a) execute the command on a workstation, (b) make an outbound request to an attacker URL (SSRF), (c) mutate a lease/session/posture or call the oracle to widen a Rust deny, or (d) leak the echoed string to a victim's Slack channel. WHY IT FAILS: handleInteraction (relay.go:212-252) has no privileged side effects whatsoever. It increments an in-memory counter, writes one FIXED-STRING log line ("interaction acknowledged" — no user input, so no log forgery), and echoes payload.command() back IN THE HTTP RESPONSE BODY to the requester. It makes zero outbound requests (enforced by the SSRF regression test at relay_test.go:143-168), never touches os/exec, session, ledger, lease, core, or the oracle (grep-confirmed: no such references in pkg/relay). The echoed command is reflected only to whoever sent the forged POST — i.e. back to the attacker themselves — and is never rendered to a victim (only genuinely Slack-originated interactions render in the channel) and never reaches a workstation (the human who clicks must manually copy/paste and run it; there is no machine path from interaction-response to execution). A forged interaction therefore yields the attacker their own input back: zero privilege gain, zero exfil, zero control-plane effect.

**Mitigation (required to ship safely):** Not required for safety (the endpoint is already side-effect-free), but if encoded as a hard gate the verification must: (1) use crypto/hmac + crypto/sha256 + encoding/hex with hmac.Equal for constant-time compare — std-lib only, honoring non-negotiable #1; (2) enforce Slack's X-Slack-Request-Timestamp skew tolerance (reject > 5 min) for spec-correct replay protection; (3) verify over the RAW request bytes (before url.ParseQuery) and ensure the maxInteractionBytes 64KB cap never truncates the signed body; (4) remain strictly default-off so absent SIR_SLACK_SIGNING_SECRET behavior is byte-identical to today (always ack 200). 401 must fire ONLY when the secret is set AND the signature is bad/missing.

**Reasoning:** The change is purely additive and default-off, so it loosens nothing and cannot widen a Rust deny, weaken fail-closed, or leak a secret — none of CLAUDE.md non-negotiables 1-7 are touched (the decision path / oracle is never invoked by this handler). Ruling out must-keep-block: there is no concrete exfil/RCE/control-plane/deny-widening path through /slack/interactions to preserve, because the handler is a pure echo with no side effects (verified: relay.go:212-252; no session/ledger/lease/exec/oracle refs in pkg/relay; SSRF foreclosed by relay_test.go:143). The "always-ack-200" rule is an availability invariant ("never wedge Slack"), not a security non-negotiable; overriding it for forged-and-only-when-opted-in requests has at worst an operator-controlled availability cost (a misconfigured signing secret 401s their own legit Slack). The signature is defense-in-depth, not load-bearing for safety, so safe-to-downgrade is the honest verdict; downgrade-with-mitigation would be defensible only if one wants the correctness conditions above encoded as a hard gate.

CRITICAL CAVEAT FOR THE SPEC (does not change THIS verdict, but the spec is misleading without it): SIR_SLACK_SIGNING_SECRET authenticates only Slack-origin requests to /slack/interactions — the HARMLESS endpoint. It does NOT authenticate the relay. The actual confused-deputy exfil surface is the SIBLING route POST /v1/detections (relay.go:160-182), which is COMPLETELY UNAUTHENTICATED: it json.Unmarshals an attacker-supplied telemetry.SlackEvent and calls r.ingest -> r.forward(blockKitMessage(ev)). blockKitMessage (blocks.go:15-56) renders ev.Title, ev.What, ev.Why, ev.Target, and ev.SuggestedActions[].Command DIRECTLY into the outbound Slack webhook payload. So a prompt-injected workstation agent — or anyone who can reach the relay's network address — can POST a forged detection whose Title/What/Command fields embed a stolen secret, and the relay will forward it straight to the operator's Slack channel where the attacker (if a channel member) or a downstream sink reads it. THIS is the #1 exfil/spoofing gap and it is a DIFFERENT trigger, entirely untouched by Slack signing (different requester — it needs a workstation-to-relay bearer token or HMAC, plus a rate limit). The implementer/operator must be told explicitly that adding SIR_SLACK_SIGNING_SECRET does NOT secure the relay's ingest path, or they will gain false confidence. That gap should be tracked and signed off as its own item (matches the verified #1 hardening finding: pkg/relay has zero authentication on /v1/detections, /slack/interactions, /healthz, /stats).
